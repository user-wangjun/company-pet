# 气泡式陪伴聊天设计

日期：2026-06-28

状态：待用户审阅

## 1. 背景与目标

愈心桌宠已有宠物气泡、宠物包台词、点击/拖拽/提醒交互和多宠物包结构。本次目标是在不把产品变成大聊天软件的前提下，让用户可以直接和当前桌宠进行轻量陪伴型对话。

第一版重点是“桌宠本体的聊天手感”：

- 通过右键双击宠物进入聊天模式。
- 在宠物旁边以连续小气泡展示用户和宠物的对话。
- 输入条稳定停在气泡聊天区底部。
- 支持停止当前回复，并把刚发送的文字撤回到输入框。
- 默认使用本地陪伴回复，架构预留后续 AI provider。
- 支持结构化用户偏好记忆，用于称呼、语气、提醒和照顾方式。
- 每只 pet 的语气、开场拟声词和本地回复内容归属自己的宠物包。

本功能不引入平台大聊天窗口，不要求账号、云同步或完整聊天历史。

## 2. 产品规则

### 2.1 入口

- 非聊天模式下，右键单击宠物只阻止系统菜单，不触发明显动作。
- 非聊天模式下，右键双击宠物进入气泡聊天模式。
- 进入聊天模式后，当前 pet 先显示一条开场拟声词气泡，例如小橘的“喵？”。
- 右键双击不再触发检查更新。
- 检查更新保留为后台或平台入口能力，不占用宠物本体手势。

### 2.2 气泡流

- 用户每次发送后新增一条用户气泡。
- 宠物每次回应后新增一条宠物气泡。
- 用户和宠物消息不挤在同一个大气泡里。
- 聊天展示区按高度限制，不按固定消息条数限制。
- 超过展示区高度后，旧气泡向上滚出或淡出。
- 长文本自动换行，不允许撑爆聊天展示区或遮挡输入条。
- 输入条固定在展示区底部，始终稳定可见。

### 2.3 停止与撤回

聊天模式下，左键单击宠物执行 `stop`，不触发普通单击交互。

`stop` 的行为：

- 如果宠物正在生成或播放回复，立即停止当前回复。
- 移除刚发出的用户气泡。
- 将刚发出的用户文字撤回到输入框。
- 输入框重新聚焦，方便用户修改后再次发送。
- 非聊天模式下，左键单击保持现有宠物单击交互。

### 2.4 退出

聊天模式支持三种退出方式：

- 按 `Esc` 立即退出。
- 点击聊天区域和宠物以外的空白处退出。
- 长时间空闲自动退出，第一版建议约 90 秒。

自动退出必须满足：

- 输入框中有未发送文字时不自动退出。
- 正在生成回复时不自动退出。
- 用户重新输入、发送、停止或聚焦输入框后重置空闲计时。

退出聊天模式后，清空临时聊天气泡并恢复普通待机气泡。

## 3. 回复来源

### 3.1 第一版默认本地回复

第一版默认使用本地陪伴回复，保证离线可用、体验稳定、实现风险低。

本地回复用于：

- 进入聊天模式后的开场拟声词。
- 用户发送后的简短陪伴回应。
- 网络或 AI provider 不可用时的降级回复。

回复语气保持短、轻、陪伴型，不输出长篇建议，不替代专业咨询。

开场气泡只使用 pet 特有拟声词，不使用完整问候句。完整句子留给宠物后续回复。

### 3.2 预留 AI Provider

平台建立统一聊天 provider 接口：

```ts
type CompanionChatProvider = {
  send(input: CompanionChatRequest): Promise<CompanionChatResponse>;
  stop(requestId: string): void;
};
```

第一版实现 `localCompanionChatProvider`。后续接入 AI 时新增远端 provider，不重写气泡 UI、聊天状态机或宠物包人格配置。

AI provider 的后续设计必须覆盖密钥、网络错误、隐私、停止生成、超时和内容安全。本次只保留接口边界，不实现真实联网对话。

## 4. 用户偏好记忆

### 4.1 目标

用户偏好记忆用于让桌宠更懂用户的陪伴方式，但不保存完整聊天记录。

第一版保存结构化偏好，而不是原始聊天内容：

- 用户称呼，例如“以后叫我阿星”。
- 语气偏好，例如“回答短一点”“不要太严肃”。
- 陪伴风格，例如“我喜欢安静陪伴”。
- 提醒偏好，例如“睡前别催太急”“喝水提醒少一点”。
- 照顾偏好，例如“最近眼睛容易累，提醒我多休息眼睛”。
- 宠物关系偏好，例如用户对某只 pet 的专属称呼或互动偏好。

### 4.2 记忆来源

偏好有两种写入来源：

- 用户主动挑明偏好时直接记住。
- 系统从聊天中自动筛选明显、低风险、稳定的偏好。

不使用确认弹窗，不逐条询问“是否记住”，避免打断陪伴感。

只要写入一条偏好，宠物必须用轻量气泡反馈，并说明记住了什么，例如：

- “好呀，我以后叫你阿星。”
- “嗯，我记住啦，以后我尽量说短一点。”
- “我记住啦，会多提醒你休息眼睛。”

用户可以通过“别记这个”“忘掉这个”等自然语言删除刚记下的偏好。

### 4.3 健康相关边界

健康内容允许作为照顾偏好保存，但不得作为医疗档案保存。

可以记：

- 作息习惯。
- 护眼、喝水、吃饭、睡觉提醒偏好。
- 用户主动表达的照顾需求。
- 与健康提醒相关的语气和频率偏好。

不自动记：

- 疾病诊断。
- 用药、病史或检查结果。
- 精确身体数据。
- 心理健康高风险内容。
- 任何需要专业判断的健康结论。

例如用户说“我有偏头痛，提醒我少盯屏幕”，可保存为“用户希望减少长时间盯屏幕并多休息眼睛”，不得保存为“用户患有偏头痛”。

### 4.4 存储与管理

用户偏好属于平台和用户，不属于某个 pet 包。建议使用版本化本地存储键：

```ts
const COMPANION_PREFERENCES_STORAGE_KEY = "yuxin-companion-preferences-v1";
```

偏好分为：

- 全局偏好：称呼、语气、陪伴风格、通用提醒偏好。
- pet 关系偏好：只对某只 pet 生效的昵称、互动或关系设定。

平台需要提供查看、编辑、删除偏好的入口。存储失败时聊天仍继续工作，只是不保留新偏好，并记录可诊断警告。

## 5. 宠物包边界

宠物特定聊天内容继续归属 `public/pets/<pet-id>/`，平台不得通过 `activePetId` 分支硬编码具体语气。

建议在 pet 包中增加可选聊天配置文件：

```text
public/pets/<pet-id>/
  pet.json
  dialogues.json
  companion-chat.json
```

`pet.json` 增加可选相对路径：

```json
{
  "companionChatPath": "companion-chat.json"
}
```

`companion-chat.json` 建议结构：

```json
{
  "openers": [
    {
      "text": "喵？",
      "sound": "chatOpenMew",
      "weight": 2
    },
    {
      "text": "mew~",
      "sound": "chatOpenMewSoft",
      "weight": 1
    }
  ],
  "localReplies": ["嗯，我听着。", "慢慢说，我陪你。"],
  "style": {
    "tone": "quiet-companion",
    "maxReplyLength": 36
  }
}
```

约束：

- 所有路径必须相对 pet 包目录。
- 文件缺失时使用平台中性陪伴回复。
- 单个 pet 缺少聊天配置不得影响其他 pet。
- `openers` 必须是短拟声词或短叫声，不使用完整问候句。
- `openers[].sound` 可选，用于映射 `pet.json` 中的 sound cue；第一版没有音频也必须能只显示文字。
- 开场声音必须短、低频、非循环，并保持 interaction-driven。
- 后续 AI prompt、角色禁忌和语气规则也应放在 pet 包内，而不是平台代码里。

## 6. 平台架构

### 6.1 聊天状态模块

新增独立聊天状态模块，负责：

- 聊天模式是否开启。
- 当前输入内容。
- 当前可见消息列表。
- 最近一次已发送但可撤回的用户消息。
- 当前回复请求 id。
- 空闲退出计时。
- `send`、`stop`、`exit` 的纯状态转移。

该模块不依赖 Pixi、Tauri 或 DOM，便于单元测试。

### 6.2 气泡聊天组件

新增气泡聊天组件，负责：

- 有限高度的消息展示区。
- 用户气泡和宠物气泡布局。
- 底部稳定输入条。
- 回车发送、按钮发送和输入聚焦。
- 长文本换行和展示区溢出处理。

组件不负责生成回复，只通过回调发出用户输入和停止请求。

### 6.3 App 集成

`DesktopPetApp` 负责：

- 将右键双击从检查更新改为进入聊天模式。
- 聊天模式下拦截左键单击，调用 `stop`。
- 非聊天模式继续保留原有单击、双击、拖拽、悬停、提醒和桌面图标交互。
- 切换 pet 时关闭当前聊天模式，避免旧宠物的气泡和人格残留。
- 加载当前 pet 的聊天配置。
- 加载用户偏好并注入当前聊天 provider 的输入上下文。

### 6.4 偏好记忆模块

新增独立偏好记忆模块，负责：

- 解析用户明确表达的偏好。
- 筛选低风险自动偏好。
- 过滤不应保存的敏感或医疗结论。
- 生成可展示给用户的“我记住了什么”反馈文案。
- 读写版本化本地偏好存储。
- 删除最近写入的偏好。
- 提供查看、编辑、删除所需的纯函数。

该模块不依赖 Pixi、Tauri 或 DOM。

### 6.5 更新检查调整

右键双击不再调用检查更新。

检查更新保留：

- 托盘菜单或平台入口手动检查。
- 后续可加低打扰后台检查。

后台发现更新时可以使用普通气泡或平台提示，但不得抢占聊天模式输入。

## 7. 数据流

1. 用户右键双击宠物。
2. 平台进入聊天模式，加载当前 pet 的开场拟声词 cue。
3. 气泡展示区显示宠物开场拟声词气泡，存在 sound cue 时播放对应短音频，输入条获得焦点。
4. 用户输入文字并发送。
5. 平台新增用户气泡，记录该消息为可撤回消息。
6. 平台调用当前聊天 provider。
7. provider 返回本地陪伴回复。
8. 平台对用户输入和回复进行偏好筛选。
9. 如果写入新偏好，平台新增一条轻量宠物反馈气泡。
10. 平台新增宠物回复气泡，清除当前可撤回状态。
11. 用户继续输入，气泡展示区按高度自然滚动。
12. 用户按 `Esc`、点击外部或空闲超时后退出聊天模式。

如果第 6 步到第 10 步之间用户左键单击宠物：

1. 平台调用 provider `stop`。
2. 移除刚发出的用户气泡。
3. 将刚发送的文字放回输入框。
4. 输入框重新聚焦。

## 8. 错误处理

- 聊天配置文件加载失败：记录警告，使用平台中性开场拟声词和本地回复。
- 聊天配置 JSON 格式无效：视为加载失败，不影响宠物基础交互。
- provider 返回失败：显示一句短降级回复，例如“刚刚有点没听清，再说一次好吗？”。
- provider 超时：停止等待，允许用户重新发送。
- 用户发送空白输入：不创建气泡，不调用 provider。
- 切换 pet、关闭应用或退出聊天模式时：取消当前回复请求和空闲计时。
- 输入框有未发送文字时：不自动退出，不清空输入。
- 偏好存储失败：聊天继续，当前偏好不持久化，并记录警告。
- 偏好筛选结果不确定：不保存，不反馈。
- 用户要求忘记刚保存的偏好：删除该偏好并给出轻量反馈。

## 9. 测试与验收

### 9.1 单元测试

覆盖：

- 右键第一次点击只 armed，第二次点击进入聊天模式。
- 右键双击不再触发检查更新。
- 聊天模式下左键单击执行 `stop`，非聊天模式下保留原单击行为。
- 发送空白输入不会新增消息。
- 发送用户消息后记录可撤回状态。
- `stop` 会移除刚发送用户气泡并把文字放回输入框。
- provider 成功回复后新增宠物气泡并清除可撤回状态。
- `Esc`、点击外部、空闲超时退出聊天模式。
- 输入框有未发送文字时不会空闲退出。
- 切换 pet 时关闭聊天模式并清理旧消息。
- 聊天配置路径使用 pet 包相对路径解析。
- 配置加载失败时使用中性降级回复。
- 开场拟声词 cue 不参与偏好检测，也不保存为聊天记录。
- 开场拟声词存在 sound cue 时会请求播放对应短音频。
- 明确偏好会写入结构化偏好并反馈记住内容。
- 自动筛选偏好只保存允许的低风险偏好。
- 健康相关输入只保存照顾偏好，不保存疾病诊断。
- “忘掉这个”会删除最近写入的偏好。
- 偏好存储失败不影响聊天发送和回复。

### 9.2 组件测试

覆盖：

- 聊天模式显示气泡展示区和底部输入条。
- 输入条发送后清空。
- 展示区高度受限，长消息不会撑爆布局。
- 多条消息超过高度后旧消息不遮挡输入条。
- 输入框聚焦后可直接打字。
- `stop` 后输入框恢复刚发送文字。

### 9.3 回归验证

实现完成后运行：

```powershell
npm test
npm run build
```

同时人工确认：

- 普通单击、双击、拖拽、提醒、悬停和桌面图标交互仍正常。
- 右键菜单不会弹出。
- 右键双击能稳定进入聊天模式。
- 检查更新仍可从保留入口触发。
- 切换 pet 后聊天语气不会串到上一只 pet。
- 用户偏好可查看、编辑、删除。
- 健康照顾偏好能影响后续提醒语气或频率。
- 浏览器预览中不因 Tauri API 缺失崩溃。

## 10. 非目标

- 不实现平台大聊天窗口。
- 不实现真实 AI API 调用。
- 不实现账号、云同步或跨设备聊天历史。
- 不保存完整聊天记录或原始长期对话。
- 不把健康偏好保存为医疗档案。
- 不做语音输入或语音朗读。
- 不改变宠物素材、动画图集或声音资源。
- 不把 pet 专属人格硬编码到 `src/`。
