# 首次欢迎信与信箱设计

日期：2026-06-19

状态：已确认

## 1. 背景与目标

愈心桌宠需要在用户首次使用时，以一封官方来信说明产品的创作初衷：
桌宠无法消除现实中的身体疲劳，但希望它能在某个平凡瞬间让用户轻松一笑。

本次目标包括：

- 首次启动平台时展示一封未拆开的官方欢迎信。
- 用户点击界面任意位置后，播放拆信动画并展开信纸。
- 用户收起信件时，播放完整的收信入箱动画。
- 在平台右上角增加信箱，允许用户以后重新阅读欢迎信。
- 建立可扩展的信件数据结构，后续更新公告可直接进入同一信箱。
- 支持批量标记已读和批量删除已读信件。
- 永久保留官方欢迎信，不允许删除。

本功能属于平台能力，不归属于任何单独桌宠包。

## 3. 首次使用流程

### 3.1 首次展示

首次启动愈心桌宠时：

1. 平台正常打开。
2. 平台内容上方覆盖半透明暗色遮罩。
3. 中央显示一只纯白、横向、现代简洁的纸质信封。
4. 信封保留清晰的三角封口。
5. 封口尖端附近显示红色火漆印，印章文字为“愈”。
6. 页面提示用户点击任意位置拆信。

整个遮罩区域均为拆信触发区。首次状态下，点击界面内任意位置都会拆信，
不要求用户精确点击信封或火漆印。

### 3.2 拆信

拆信动画按以下顺序执行：

1. 红色火漆印淡出。
2. 三角封口向上翻开。
3. 信纸从信封顶部开口升起。
4. 信纸放大并移动到平台窗口正中央。
5. 信件正文淡入。

信件实际展开后，立即将该信标记为已读。只有完成拆信才算用户已经阅读过
首次欢迎信。

如果用户没有拆信便退出应用，下次启动仍然显示未拆信封。

### 3.3 阅读与关闭

信纸展开后：

- 点击信纸正文不会关闭。
- 点击信纸外的遮罩区域会收信。
- 点击信纸右上角关闭按钮也会收信。
- 两种关闭方式使用完全相同的收信动画和状态处理。

### 3.4 收信入箱

收信不能直接隐藏信纸，也不能将信纸压缩成一个独立方块飞走。

必须按以下空间关系执行：

1. 信纸在信封上方先完成折叠。
2. 折叠后的信纸停在信封顶部开口上方。
3. 信纸从顶部开口向下插入信封。
4. 信纸完全进入信封，不得从顶部或底部突出。
5. 三角封口向下合上。
6. 红色火漆印重新出现。
7. 完整信封缩小并飞向平台右上角的信箱按钮。
8. 信箱按钮轻微弹动，表示信件已经收好。
9. 遮罩关闭，恢复平台操作。

## 4. 信纸与字体视觉

### 4.1 信纸

- 信纸为浅色纸张，保持高对比度。
- 展开后位于平台窗口正中央。
- 尺寸接近平台主要可用阅读区域，左右保留约 40 像素安全边距。
- 在当前平台窗口尺寸下，无需滚动即可完整显示欢迎信。
- 信纸右上角提供清晰但不抢眼的关闭按钮。

### 4.2 字体

- 标题、正文和署名均使用行楷字体。
- 字体回退顺序优先使用 `STXingkai`、`华文行楷`，再回退到楷体和衬线字体。
- 不使用浏览器伪粗体，以免笔画粘连、发糊。
- 正文使用正常字重和深墨色高对比文本。
- 参考尺寸为正文 22 像素、标题 34 像素。
- 正文行高约为 1.65，段落之间保留明确间距。
- 正文保持适当左右页边距，避免单行过长或文字拥挤。

如果目标系统不存在行楷字体，使用楷体回退，不能因字体缺失导致布局溢出。

## 5. 信箱界面

### 5.1 入口

平台右上角增加信封图标按钮，作为信箱入口。

- 存在未读信件时显示未读数量。
- 没有未读信件时不显示数字徽标。
- 信箱入口只在平台界面中提供，本次不增加右键菜单入口。

### 5.2 信件列表

打开信箱后显示信件列表，而不是直接打开欢迎信，以便以后加入更多官方来信
和更新公告。

每封信至少显示：

- 标题
- 发布者或署名
- 发布日期
- 信件类型
- 已读或未读状态
- 永久保留标记

点击列表项后打开对应信纸。通过信箱打开信件时直接展开信纸，不重新播放首次
未拆信封的强制引导。

### 5.3 批量操作

信箱提供：

- 一键已读：将当前所有未读信件标记为已读。
- 删除已读：删除所有允许删除且已经阅读的信件。

《致初次相遇的您》是永久信件：

- 永远显示在信箱中。
- 不显示单封删除入口。
- 不参与“删除已读”。
- 即使已经阅读也不可删除。

如果没有可删除的已读信件，“删除已读”按钮应禁用，不能显示误导性的成功状态。

## 6. 信件数据模型

平台使用统一信件定义，以支持欢迎信和后续更新公告：

```ts
type PlatformLetter = {
  id: string;
  title: string;
  sender: string;
  publishedAt: string;
  type: "welcome" | "update";
  paragraphs: string[];
  permanent: boolean;
};
```

首封欢迎信使用稳定 ID，例如：

```ts
const WELCOME_LETTER_ID = "welcome-first-meeting";
```

欢迎信的 `publishedAt` 固定为 `"2026-06-19"`。

信件 ID 一旦发布便不得修改。已读和删除状态通过 ID 关联；修改 ID 会使老用户
重新收到同一封信。

本次将内置信件定义保存在平台代码的独立信件模块中。后续增加更新公告时，
只需新增信件数据，不应重写首次欢迎信状态逻辑。

## 7. 状态与持久化

### 7.1 持久化状态

使用 `localStorage` 保存：

```ts
type MailboxState = {
  readLetterIds: string[];
  deletedLetterIds: string[];
};
```

本地存储键固定为 `yuxin-mailbox-state-v1`。如果以后改变结构，使用新的版本化
存储键或显式迁移逻辑，不能静默改变现有值的含义。

首次欢迎信是否自动展示由 `readLetterIds` 决定：

- 不包含欢迎信 ID：首次启动遮罩继续出现。
- 包含欢迎信 ID：后续启动不再自动展示。

不单独保存“展示过”状态，避免用户尚未拆信就永久错过欢迎信。

### 7.2 状态规则

- 拆开信件时写入已读状态。
- 通过信箱打开未读信件时也写入已读状态。
- 一键已读写入所有当前可见信件 ID。
- 删除已读只写入非永久信件 ID。
- 永久信件即使错误出现在删除状态中，也必须在读取时恢复显示。
- 不存在于当前信件目录中的旧 ID 可以保留，不影响渲染。

### 7.3 存储失败

读取或写入 `localStorage` 失败时：

- 平台和桌宠功能继续工作。
- 当前运行期间的已读、删除状态仍在内存中生效。
- 记录可诊断警告。
- 不弹出错误对话框打断用户。
- 下一次启动可能重新显示欢迎信，这是可接受的降级行为。

## 8. 组件与代码边界

正式实现应拆分以下职责：

### 8.1 信件领域模块

负责：

- 信件数据类型和内置信件目录。
- 欢迎信稳定 ID。
- 已读和删除状态的纯函数更新。
- 永久信件删除保护。
- 未读数量和可见信件筛选。
- `localStorage` 的安全读取和写入。

该模块不得依赖 Pixi、Tauri 窗口或宠物交互代码。

### 8.2 信箱界面

负责：

- 右上角信箱按钮和未读徽标。
- 信件列表。
- 一键已读和删除已读按钮。
- 打开指定信件。

### 8.3 信件阅读器

负责：

- 首次未拆信封状态。
- 拆信、展开、关闭和收信入箱动效。
- 信纸版式。
- 首次展示与从信箱阅读两种入口。

### 8.4 平台整合

`DesktopPetApp` 只负责：

- 读取当前信箱状态。
- 控制信箱或信件阅读器是否打开。
- 将读信、批量已读和删除操作交给信件模块。
- 保持现有宠物加载、气泡、窗口布局和平台卡片功能不变。

不将信件业务规则直接硬编码为大量 `App.tsx` 条件分支。

## 9. 交互冲突与可访问性

- 信件遮罩打开时，平台卡片和桌宠选择不可点击。
- 信箱或信件阅读器中的点击事件不得触发平台窗口拖动。
- 打开信件时将焦点移动到信件阅读器。
- 关闭按钮具备明确的中文无障碍名称。
- `Escape` 键关闭已展开信件，并触发正常收信动画。
- 动画进行期间忽略重复关闭操作，避免重复写状态或创建多个飞行动画。
- 对启用“减少动态效果”的系统，缩短或简化动画，但保持相同状态流转。

## 10. 测试与验收

### 10.1 状态单元测试

覆盖：

- 新用户看到欢迎信。
- 未拆信退出后仍属于未读状态。
- 拆信后欢迎信进入已读列表。
- 后续启动不再自动显示已读欢迎信。
- 欢迎信始终保留，不能被单独或批量删除。
- 一键已读标记全部当前可见信件。
- 删除已读只删除非永久信件。
- 未读数量排除已读和已删除信件。
- 损坏或无效的本地存储数据安全回退。
- 写入失败时当前内存状态仍可继续使用。

### 10.2 组件测试

覆盖：

- 首次信封遮罩出现。
- 点击遮罩后信纸展开。
- 点击正文不会关闭。
- 点击外部区域或关闭按钮进入收信状态。
- 动画期间重复关闭不创建第二次收信。
- 信箱按钮显示正确未读数量。
- 信箱列表展示永久标记。
- 无可删除信件时删除按钮禁用。
- 从信箱重新打开欢迎信时不显示首次拆信提示。

### 10.3 视觉验收

- 信封为白色横向现代信封，不使用米黄色复古风格。
- 红色火漆印位于三角封口尖端附近。
- 信纸在窗口中央展开。
- 正文行楷清晰，无伪粗体造成的笔画粘连。
- 正文无需滚动即可完整阅读。
- 收信时信纸先在上方折叠，再从顶部开口进入信封。
- 封口前信纸上下均无突出或阴影溢出。
- 完整信封飞入右上角信箱，信箱有明确反馈。

### 10.4 回归验证

实现完成后运行：

```powershell
npm test
npm run build
```

同时人工确认：

- 宠物选择仍正常工作。
- 平台窗口拖动与关闭仍正常工作。
- 桌宠气泡、动画和声音不受影响。
- 现有宠物包结构和资源路径未被修改。
